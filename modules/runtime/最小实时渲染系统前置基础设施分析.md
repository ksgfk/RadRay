# 最小实时渲染系统前置基础设施分析

## 1. 目的

本文档用于回答一个更现实的问题：

在 [实时渲染系统MVP架构设计](C:/Users/xiaoxs/Desktop/RadRay/modules/render/runtime/实时渲染系统MVP架构设计.md) 已经确定总体分层之后，如果要把这套设计真正落地成“能跑起来”的最小实时渲染系统，在工程内还必须先补齐哪些前置基础设施。

这里关注的不是最终理想形态，而是：

- 哪些能力当前工程已经具备
- 哪些能力是 MVP 真正缺失的前提条件
- 这些前提条件之间的依赖关系是什么
- 应该按什么顺序补，才能最快形成闭环

## 2. 当前工程现状

结合当前仓库代码，可以先做一个明确判断：

**现有 `modules/render` 已经具备较完整的 RHI 原语，但尚未建立运行时中层。**

也就是说，目前已经有：

- `Device`
- `CommandQueue`
- `CommandBuffer`
- `SwapChain`
- `Buffer` / `Texture`
- `BufferView` / `TextureView`
- `Fence`
- `Shader`
- `RootSignature`
- `GraphicsPipelineState` / `ComputePipelineState`
- 显式 `ResourceBarrier`

这些底层能力足以证明：

- 后端不是空白
- 可以创建 swapchain 并完成最小的清屏 + present
- 可以手工录制命令和 barrier

从现有测试也能看出，工程已经可以直接手写一条底层渲染路径，例如：

1. 创建 device / queue / command buffer
2. 创建 swapchain
3. acquire backbuffer
4. 手工插入 `RenderTarget -> Present` 资源状态转换
5. begin render pass
6. submit
7. present

这说明 **MVP 的真正缺口不在 RHI backend，而在 RHI 之上的 runtime / resource / graph / pipeline 层。**

换句话说，当前工程更像是：

- 有底层 GPU API 抽象
- 有少量 GPU 资源辅助工具
- 没有正式的渲染运行时骨架

## 3. 当前真正缺失的层

若按架构设计文档中的五层划分：

1. Simulation
2. Extract
3. Render Pipeline
4. Render Graph
5. RHI Backend

那么当前工程主要只覆盖了第 5 层的一大部分，以及少量第 4 层以下的基础类型。

真正还不存在的，是以下中层能力：

- 逻辑世界到渲染世界的正式同步边界
- 渲染资源的正式身份系统
- 资源生命周期管理
- 帧内与跨帧内存管理
- 图描述、图编译、图执行
- 渲染线程运行时壳层
- 资产上传与退休回收系统

因此，实现 MVP 的重点不是“再补一个底层 API”，而是建立 **中层基础设施**。

## 4. 最核心的前提：先拆清楚 5 类生命周期

在真正写代码前，必须先把以下五类对象的生命周期从概念上拆开：

### 4.1 逻辑世界生命周期

包括：

- ECS 组件
- CameraState
- MeshInstanceState
- LightState
- MaterialInstanceState

它们属于 Simulation，不允许被渲染线程直接跨线程访问。

### 4.2 FrameSnapshot 生命周期

`FrameSnapshot` 是 CPU 线程之间的数据交接物。

它：

- 在逻辑线程构建
- 在渲染线程消费
- 在 CPU 消费完成后即可回收
- 不需要活到 GPU 执行结束

### 4.3 GPU Inflight Frame 生命周期

这类对象用于管理 GPU 执行中的帧上下文，例如：

- command buffer
- fence
- upload arena
- descriptor arena
- frame-local allocator

它们必须等 GPU retire 后才能重置或回收。

### 4.4 长寿命渲染资产生命周期

包括：

- 静态 mesh GPU buffer
- 静态 texture
- shader
- material GPU 参数布局
- PSO cache

它们由资源系统持有，不属于单帧对象，也不属于 graph transient。

### 4.5 Render Graph 资源生命周期

包括：

- external
- transient
- persistent

这是 graph/compiler 处理的资源生命周期，不能直接等同于资产生命周期或 inflight frame 生命周期。

### 4.6 为什么这一步必须最先明确

因为一旦这五类生命周期混在一起，后面会立刻出现以下问题：

- `FrameSnapshot` 被错误绑到 GPU fence 上，导致逻辑线程被 GPU 拖慢
- 静态资源和 transient 资源混成同一套分配器，难以回收
- render thread 回头访问逻辑对象，破坏线程边界
- pass 直接 new 底层资源，graph 失去生命周期控制

因此，**实现 MVP 的第一个前置条件，不是 RenderGraph 本身，而是生命周期边界稳定。**

## 5. 必须先有的前置基础设施

下面这些能力，是实现一套最简实时渲染系统时真正绕不开的基础设施。

---

## 6. 资源身份系统：RenderAssetRegistry

### 6.1 为什么它是第一优先级

`FrameSnapshot` 文档明确要求：

- 不持有 GPU 对象
- 不持有逻辑对象裸指针
- 只持有稳定资源句柄

这意味着必须先定义一套正式的“资源身份层”。

### 6.2 至少要有的句柄

MVP 阶段至少需要：

- `MeshHandle`
- `MaterialHandle`
- `TextureHandle`
- `BufferHandle`
- `ShaderHandle`
- `SamplerHandle`

必要时还可以补：

- `PipelineHandle`
- `RenderViewHandle`

### 6.3 资源身份层需要解决的问题

这层至少要能回答：

- 某个 mesh handle 对应哪些 vertex/index buffer
- 某个 material handle 对应哪组 shader / root signature / render state
- 某个 texture handle 当前是否已上传到 GPU
- 某个资源默认 view 是什么
- 某个资源格式、尺寸、usage、初始状态是什么
- 某个资源是否属于 external / asset-owned / runtime-owned

### 6.4 如果没有这层会怎样

如果没有正式资源身份系统：

- `FrameSnapshot` 只能塞裸指针或临时对象
- Extract 层无法输出稳定的渲染引用
- Render 线程无法独立准备 draw item
- 后续 graph 也无法把“逻辑句柄”映射到“物理资源”

因此，**资源身份层是整个运行时的第一块地基。**

---

## 7. 上传系统：UploadSystem

### 7.1 为什么它是 MVP 必需项

任何“能画出东西”的实时渲染系统，都绕不过以下最基本问题：

- 顶点数据怎么进 GPU
- 索引数据怎么进 GPU
- 纹理数据怎么进 GPU
- 每帧常量数据怎么上传
- 上传完成后何时可以释放 staging 内存

### 7.2 现有工程已有但不够的部分

当前已有 `CBufferArena`，这说明工程已经具备：

- 上传堆 buffer 创建
- 常量缓冲线性分配
- Map 持久映射的基本思路

但这还不够，因为完整上传系统还需要：

- staging / upload ring
- buffer copy / texture copy 录制
- 初始资源状态管理
- 上传请求聚合
- 与 fence retire 绑定的回收

### 7.3 MVP 需要的最小上传能力

第一版至少需要支持：

1. 静态 mesh 初始化上传
2. 静态纹理初始化上传
3. 每帧 constant / structured 数据上传
4. 提交后按 inflight frame 回收 upload 内存

### 7.4 为什么不能把上传逻辑塞进 Pass

因为上传系统本质上不是某个具体渲染 pass 的私有逻辑，而是：

- 运行时级别的资源准备能力
- graph 执行前后的共享基础设施

如果每个 pass 自己管理 staging 和 copy：

- 生命周期会失控
- 回收规则会混乱
- graph 无法统一感知资源准备关系

---

## 8. GPU Inflight Frame 管理：RenderFrameContext / FrameInFlight

### 8.1 这层的职责

必须有一个正式对象管理“GPU 正在执行的帧上下文”。

建议至少包含：

- `FrameIndex`
- `CommandBuffer`
- `Fence`
- `FrameLinearAllocator`
- `UploadArena`
- `DescriptorArena`
- `CBufferArena` 或每帧常量缓冲分配器

### 8.2 为什么它不能等同于 FrameSnapshot

`FrameSnapshot` 是 CPU 可见性交接对象；
`FrameInFlight` 是 GPU 执行与回收对象。

这两者如果强行绑在一起，会导致：

- CPU 快照释放被 GPU fence 拖住
- 快照队列吞吐下降
- 逻辑线程容易阻塞

因此必须正式拆分。

### 8.3 MVP 需要解决的问题

这层至少要定义清楚：

- 每帧使用哪个 command buffer
- 何时 `Fence::Wait`
- 何时可 reset/upload/descriptor arena
- backbuffer index 和 inflight frame index 的关系
- 哪些资源跟随 inflight frame 回收

没有这层，后面所有 graph execute、upload retire、descriptor reset 都没有稳定落点。

---

## 9. 快照构建与交接：FrameSnapshotBuilder + FrameSnapshotQueue

### 9.1 为什么必须是正式设施

文档已经把 `FrameSnapshot` 定义为逻辑线程和渲染线程之间唯一正式同步边界，因此它不能只是一个普通 struct。

真正可用的实现必须至少包括：

- `FrameSnapshotBuilder`
- 每 slot 自有 arena
- `Finalize()`
- `FrameSnapshotQueue`
- 发布协议
- 回收协议
- 调试校验

### 9.2 Queue 不能直接拿现有 Channel 顶替

工程内已有 `BoundedChannel`，但它不能直接替代 `FrameSnapshotQueue`。

原因在于 `FrameSnapshotQueue` 还要求：

- 固定槽位复用
- `Free / Building / Published / Reading` 状态
- 快照自有存储区
- latest complete snapshot 策略
- 可丢弃旧 published snapshot

这些都超出了通用 channel 的职责。

### 9.3 Builder 必须解决的问题

`FrameSnapshotBuilder` 至少要保证：

- 所有 span 指向 slot 自己拥有的内存
- `UploadRequest::Data` 指向有效稳定区域
- finalize 后快照不可变
- Header count 与数组长度一致
- handle、view id、camera id 等引用关系合法

如果没有 builder 和 finalize 约束，快照很容易变成“逻辑容器的浅包装”，最终失去线程安全意义。

---

## 10. RenderScene / Prepare 层

### 10.1 为什么光有 Snapshot 还不够

`FrameSnapshot` 只表达渲染事实，不表达如何绘制。

但一个真正可执行的渲染 pass，需要的是更贴近 draw 的结构，例如：

- `PreparedView`
- `PreparedDrawItem`
- `PreparedBatch`
- `PreparedLightList`

也就是说，在渲染线程上还需要一个“Prepare”阶段。

### 10.2 这层的最小职责

MVP 阶段即使不做复杂优化，也建议这层至少负责：

- 按 view 过滤对象
- 解析 mesh / submesh / material handle
- 构建 draw item
- 基于 sort key 做基础排序
- 生成 per-pass 可直接遍历的数据结构

### 10.3 为什么不能让 Pass 直接啃原始 Snapshot

如果 pass 直接遍历 `FrameSnapshot::MeshBatches` 并现场解析：

- pass 代码会承担过多 scene 逻辑
- pipeline 和 scene 边界会混乱
- 后续裁剪、合批、实例化很难集中演进

因此建议单独保留 `RenderScene` 或 `Prepare` 层。

---

## 11. 材质契约与 PSO 系统

### 11.1 这是“能发 draw call”的真正前提

最小实时渲染系统想要画出一个 mesh，不只是有 vertex/index buffer 就够了，还必须回答：

- 用哪组 shader
- 用哪种 vertex layout
- 根签名/绑定布局是什么
- 常量参数怎么组织
- 纹理和 sampler 如何绑定
- raster/depth/blend 状态是什么

这些实际上构成了“材质契约”。

### 11.2 MVP 阶段至少要固定的内容

第一版建议直接把材质系统收缩到很小：

- 一个 forward opaque 材质模型
- 一个固定的 vertex layout
- 一套固定 root signature / binding layout
- 少量固定的材质参数槽
- 一个 PSO cache

### 11.3 为什么它必须在 Graph 前落地

因为 graph 负责组织 pass 与资源依赖，而不应该负责决定：

- 某个 draw item 应该绑定什么 shader
- 某个 material 如何映射成 PSO

这些都属于 scene/pipeline 侧的契约问题。

如果这一层未定义，`MainScenePass` 甚至无法稳定录制最基本的 draw。

---

## 12. Descriptor 管理

### 12.1 现有 RHI 提供了 view / descriptor set 原语

但 MVP 还缺一层运行时级别的 descriptor 管理：

- 每帧 descriptor arena
- 静态 descriptor cache
- material/texture/sampler 绑定更新
- 生命周期与 inflight frame 的绑定

### 12.2 为什么它是前置设施

因为只要开始真正绘制 mesh，就一定会遇到：

- 本帧临时描述符从哪里分配
- 一份 descriptor set 可复用多久
- texture view / sampler view 如何缓存
- 哪些 set 跟随 material 生命周期，哪些跟随 frame 生命周期

如果没有这层，绘制代码会迅速散落到各个 pass 中，难以收敛。

---

## 13. 最小 RenderGraph IR 与 Compiler

### 13.1 它需要解决什么

当资源系统、frame context、snapshot、prepare 都存在后，才适合引入最小 graph。

MVP graph/compiler 至少需要以下能力：

1. `ResourceHandle` / `PassHandle`
2. external resource import
3. transient resource create
4. pass 读写声明
5. 拓扑排序
6. 非法访问检测
7. 资源生命周期区间分析
8. transient 物理分配
9. barrier 生成
10. execute context 映射

### 13.2 为什么当前工程还不能直接写 Graph Execute

虽然已有 RHI barrier 原语，但现在 barrier 仍是调用者手写。

例如 swapchain test 中已经体现出：

- acquire backbuffer
- 手动记录 `Undefined/Present -> RenderTarget`
- render pass
- 手动记录 `RenderTarget -> Present`
- submit + present

这说明 graph/compiler 还没有接管：

- 资源状态追踪
- pass 间 barrier 推导
- external resource import 状态管理

### 13.3 MVP 的 graph 不需要太复杂

第一版不需要：

- aliasing
- pass merge
- multi queue
- async compute
- 可视化编辑器

但必须做到：

- 让 pass 不再手写 barrier
- 让 runtime 不再手写 transient 生命周期

---

## 14. Presentation Runtime Shell

### 14.1 为什么还需要这一层

仅有 graph 和 RHI 还不够，系统还需要一个“真正驱动一帧”的运行时壳层。

它至少要负责：

- 持有 device / queue / swapchain
- 管理 resize / recreate
- acquire backbuffer
- 创建并初始化 `RenderFrameContext`
- 调 pipeline build graph
- 调 graph compile / execute
- submit
- present
- retire inflight frame

### 14.2 它的意义

这层是把所有中层能力串起来的宿主。

没有 runtime shell，就只能得到一堆零散子系统，而不是一个可运行的渲染器实例。

---

## 15. 调试、校验与可观测性

### 15.1 这不是锦上添花，而是必要前置

实时渲染系统一旦引入：

- 多线程
- handle 间接层
- graph 编译
- inflight frame
- 上传回收

调试难度会立刻上升。

因此最小可观测性能力必须尽早加入。

### 15.2 建议最早期就支持的诊断

- snapshot finalize 校验
- handle 有效性断言
- graph 资源访问合法性断言
- 资源状态跟踪断言
- debug name 贯穿 buffer / texture / pass
- 基础帧统计
- graph dump 或文本打印

没有这些工具，后续出现问题时很难判断是：

- extract 错了
- handle 失效了
- descriptor 绑定错了
- barrier 漏了
- 还是 frame retire 提前了

---

## 16. 哪些东西不是 MVP 前置条件

为了避免过度设计，也必须明确哪些能力可以延后。

以下内容 **不是** 第一阶段必须先完成的前置基础设施：

- 通用材质节点系统
- Render Graph 可视化编辑器
- 自动 Pass 合并
- Async Compute
- 多队列调度器
- Bindless 全面资源体系
- GPU Driven
- 通用 temporal 框架
- 复杂 renderer list
- 全自动 shader reflection DSL

这些能力都可以建立在前述基础设施稳定之后逐步加入。

## 17. 建议的最小模块划分

如果要开始正式落代码，建议先把以下模块概念立住。

### 17.1 `modules/render/runtime`

放运行时骨架与帧级对象：

- `frame_snapshot.h`
- `frame_snapshot_builder.h`
- `frame_snapshot_queue.h`
- `render_frame_context.h`
- `frame_inflight.h`
- `renderer_runtime.h`
- `upload_system.h`

### 17.2 `modules/render_scene`

放 scene 到 draw-prep 的中间层：

- `render_scene_types.h`
- `prepared_view.h`
- `prepared_draw_item.h`
- `render_prepare.h`

### 17.3 `modules/render_pipeline`

放 pipeline 组织层：

- `render_pipeline.h`
- `forward_pipeline.h`
- `main_scene_pass_setup.h`
- `present_pass_setup.h`

### 17.4 `modules/render_graph`

放 graph 和 compiler：

- `render_graph.h`
- `render_graph_types.h`
- `pass_builder.h`
- `resource_registry.h`
- `graph_compiler.h`
- `graph_executor.h`

### 17.5 `modules/rhi` 是否需要单拆

当前仓库的 RHI 代码在 `modules/render` 中，MVP 阶段不一定必须立即拆模块。

更现实的做法是：

- 先保留现有底层目录不动
- 在其上补 runtime / graph / pipeline
- 等中层稳定后，再评估是否单独抽出 `modules/rhi`

## 18. 推荐实现顺序

下面给出一条更符合当前工程现状的落地顺序。

### Phase 0：先固定资源身份与生命周期模型

先定义：

- 资源句柄类型
- 资产注册表
- inflight frame 与 snapshot 的生命周期边界
- external / transient / persistent 的正式分类

这一阶段主要目标不是画图，而是定规则。

### Phase 1：资源系统与上传闭环

先实现：

- `RenderAssetRegistry`
- mesh / texture / shader / material 注册
- `UploadSystem`
- 每帧 upload arena
- retire 回收

目标：

- 能把静态 mesh、静态 texture 可靠送进 GPU

### Phase 2：单线程 Render Runtime 闭环

实现：

- `RenderFrameContext`
- 最小 runtime shell
- 单线程版 `FrameSnapshotBuilder`
- 一个极简 `FrameSnapshot`
- `ForwardPipeline`

目标：

- 先在单线程下，用正式的数据结构打通整条链路

### Phase 3：最小 MainScenePass 可绘制

实现：

- `PreparedDrawItem`
- 固定材质契约
- 固定 root signature / PSO cache
- 一个最小 forward opaque pass

目标：

- 真正画出 mesh，而不是只 clear backbuffer

### Phase 4：最小 RenderGraph

实现：

- resource import / create
- pass read/write declaration
- compile
- barrier generation
- execute

第一版链路建议只支持：

1. `UploadPass`
2. `MainScenePass`
3. `PresentPass`

目标：

- 去掉手写 barrier
- 正式形成 graph 调度骨架

### Phase 5：双线程 FrameSnapshotQueue

实现：

- slot-based `FrameSnapshotQueue`
- publish / acquire / release 协议
- latest-complete snapshot 策略

目标：

- 让逻辑线程和渲染线程正式解耦

### Phase 6：完善 transient / persistent 资源生命周期

实现：

- transient resource registry
- persistent resource registry
- 按 graph 生命周期分配和回收
- frame retire 集成

目标：

- 为后续 post process、temporal、shadow 留正式扩展点

## 19. 一个重要结论

实现一套最简实时渲染系统时，最容易犯的错误是：

> 误以为“先写一个 RenderGraph”就是开工。

实际上，真正的第一优先级应该是：

1. 资源身份稳定
2. 生命周期边界稳定
3. 上传和 inflight frame 管理稳定
4. 快照交接边界稳定

只有这几层先稳住，后续的：

- pipeline
- graph
- barrier generation
- persistent history
- post process

才有可靠的落点。

换句话说：

**MVP 的前置条件不是“更复杂的渲染功能”，而是一套最小但完整的资源与帧运行时基础设施。**

---

## 20. 建议的第一批实际代码目标

如果接下来要开始进入编码阶段，建议第一批真正落地的目标收敛为：

1. `FrameSnapshot` 最小结构与 builder
2. `RenderAssetRegistry` 最小版本
3. `UploadSystem` 最小版本
4. `RenderFrameContext` / `FrameInFlight`
5. 固定材质契约 + 一个最小 forward draw path
6. 单线程 `Upload -> MainScene -> Present` 闭环

只要这六项完成，工程就会从“有 RHI 原语”升级为“有最小可运行的实时渲染 runtime”。

这时再引入 RenderGraph，会是顺水推舟，而不是空中楼阁。
