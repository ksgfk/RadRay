# 阶段二深度落地方案：宏观 Pass 图拓扑与生命周期引擎

本阶段的任务完全解耦图形 API（无关 Vulkan/DX），是一个纯粹的 **DAG（有向无环图）处理系统**。如果本阶段的生命周期计算出锚定错误，阶段三的 Memory Aliasing 和 Barrier 计算将产生摧毁性的时序崩死。

必须以“极度严谨的数据结构”和“不可逾越的编译器契约”作为这套引擎的底层底座。

---

## 一、 虚拟资源架构设计 (Virtual Resource System)

在 Render Graph 中，资源的注册实际上是被分配了一个带有状态机的“元追踪器 (Tracker)”，这里不允许任何实体的显存直接开辟。

### 1. 结构与句柄设计

*   **`ResourceHandle`**：一种强转型的整数/位域包装类型，不允许暴露底层指针。
    设计参考：`[ Index (24-bit) | Generation / Version (8-bit) ]`。每次更新都产生新 Version 配合检测。
*   **`VirtualResource` 实体记录**：
    *   **基础属性**：资源类型 `Buffer` / `Image_2D/3D/Cube`，以及尺寸求值闭包 `Resolution (如相对于屏幕的倍数)`，`Format`。
    *   **持久特性 (Lifespan Class)**：
        *   `Transient` (单帧存活，默认项，最高级显存优化对象)
        *   `Persistent / Temporal` (跨越多帧，比如 History Buffer。这些资源直接分配持久显存池，不参与 Alias 图着色但参与 Barrier 追踪)。
        *   `External / Swapchain` (外部导入，不可修改布局或无权 Aliasing的系统级资源)。

### 2. 构建任务流：资源工厂实现

*   [ ] 设计 `RegistryPool<VirtualResource>`，确保图编译阶段任何资源注册皆 O(1) 并紧凑存在一个线性 Vector 中供后续 Cache-friendly 遍历扫描。
*   [ ] 在图的头部建立 `RegisterExternal()` 方法支持绑定诸如 Swapchain Backbuffer 的特殊追踪。

---

## 二、 图节点的构建机制 (Node Building)

这里定义开发者暴露的节点 `RGPassNode` 与内部编译器读取的状态，将声明与执行完全物理隔离。

### 1. 数据结构分离律（Setup 与 Execute 通道分离）

这是一个图系统必须死守的架构红线：**所有的图边关系必定发生在 Execute （正式提交渲染逻辑）之前**。

*   **`RGPassNode` 核心数据体**:
    *   `std::vector<ResourceAccessEdge> reads;`
    *   `std::vector<ResourceAccessEdge> writes;`
    *   `QueueType queue_class` (如 Graphics_Async, Compute_Async, Transfer)。
    *   `bool is_alive = false;` （留作第三节执行 DCE 死码清理时用）。
    *   `std::function<void(VirtualCommandContext&, ResourceVault&)> execute_callback;`

*   **`ResourceAccessEdge` (依赖连线条)**:
    用于记录这个 Pass **“将会如何以及在哪条流水线（Stage）使用什么资源”**：
    *   `ResourceHandle handle;`
    *   `AccessMode mode` (如 `SampledRead`, `ColorAttachmentWrite`, `ComputeStorageRead`)。

### 2. 构建任务流：节点挂载实现

*   [ ] 提供 Builder Pattern 包装接口：保证用户连续调用 `.read().write()` 时的安全与防呆（如重复录入和自读写回放异常检测）。
*   [ ] 提供 `Shader Parameter Struct` (类似 RDG 结构) 或自动反射（阶段一接驳处）对隐式传入的资源进行 `ResourceAccessEdge` 转换映射收集。

## 三、 死代码消除（Dead Code Elimination / Resource Culling) 

这是渲染图省去无用开销的关键（举例：当全局关闭了泛光或者镜头畸变等特效时，系统必须能聪明地斩除那些前置的 `Bloom_Downsample`, `Bloom_Blur` 的几层连环 Pass）。这一过程比排序还要早。

### 1. 逆向图遍历根源算法 (Sink-Driven Trace-Back)

我们不可臆测某个中间节点是否可以抛弃，必须从**必然保留的根节点（Root Sinks）**往上回溯。

*   **Root Sinks 判定准则**：
    1.  最终投递的 `Backbuffer / SwapchainImage`。
    2.  被显式修饰为 `.mark_output()` 或 `.force_retain()` 的中间阶段调试图。
    3.  跨帧读取并在下一个独立系统使用的资源（比如传给引擎上层 AI 或物理演算的 `ReadBackBuffer_Persistent`）。
    4.  没有连通下游，纯副作用操作或标记了 `Transfer` 不包含逻辑目标的纯搬运行为可能也需要打散成特殊护城集防误杀。

### 2. 构建任务流：DCE 实现

*   [ ] 初始化 Pass 状态图表，将所有 `RGPassNode.is_alive` 以及虚拟资源上的 `resource.is_necessary` 清零复位至 `false`。
*   [ ] 提取全部符合上述 Root Sinks 条件的 `VirtualResource` 放入处理队列 `Queue<ResourceHandle> worklist` 且打上 `is_necessary = true` 标签。
*   [ ] **BFS (广度优先反推) 流程编写**:
    *   Dequeue 资源 R。
    *   找到所有执行过 `write(R)` 的 `RGPassNode P`。若 `P` 已经标记为 `alive` 则跳过；若未标记则设 `P.is_alive = true`。
    *   遍历挂在且存活的 `P` 中的所有依赖项 `reads(T)`，将这批刚刚被赋予新生命的原材料 `T` 入队 `worklist` 重复直至队列为干。
*   [ ] 在大执行列表 Vector 中执行 `std::erase_if` 删除所有 `is_alive == false` 的僵尸 Pass 从而物理斩断一切没用的管线与绘制开销。

## 四、 图调度与拓扑有序化 (Topological Sorting)

经过清洗后剩下了全是有用的 Pass。此刻它们只是由一系列虚无边界拼成的网，我们需要把它们拍平定先后顺序（Linearize），也就是获得一个提交给 GPU 跑的真实时序轴。

### 1. Kahn 并发无环推导法优化

*   由于有些兄弟 Pass 它是**互不阻塞**（例如 SSAO 的降采样模糊 跟 贴图高光准备）。在图谱上他们入度都是 0。
*   **引擎最佳调度偏好补充 (Heuristic Weight weighting)**：当检测到同处无碍起跑线的多个节点时，应具有排序偏向，以尽量拉长某个资源的生产同使用间的间隔时间，增大异步隐去能力。例如，**始终拔高 AsyncCompute 队列相关起点的分量**，尽早发生以实现和 Graphics 的无缝拼装。

### 2. 构建任务流：拓扑化器实现

*   [ ] 构建资源级的生产者-消费者网图 `AdjacencyList`。建立双向图：`Producer -> Consumer` 和 入度表 `Indegree[PassID]`。
    根据 Pass 中的读写表生成依赖关系。注意 `Write -> Write` 以及 `Write -> Read` 皆被视为依赖强化图边界边。
*   [ ] 使用 Kahn 算法进行扫描。当处理入度为 0 的 Pass 队列时，若出现分叉，应根据引擎启发因子（如 Queue 权重）或使用稳定的按注册先后序（ID 大小）发包输出顺序序列化列表 `std::vector<Pass*> sorted_passes`。
*   [ ] **环侦测哨兵**: 若全部 Pass 录完但发现部分节点的图入度不为 0（还有挂起的死循环引用没有输出），说明用户录制产生了 `A依赖B, B读A`的致命闭环。此刻应当抛出极其严格的图引擎 `Fatal Error: Circular Dependency Detected` 并直接 Dump 当前残图文本以协助开发者分析逻辑死锁。

## 五、 宏观生命周期采集 (Lifetime Analysis)

此时的 Pass 已经具有稳定的先后物理排列 `[ 0, 1, 2, ..., N ]`。基于这个时间轴来确定资源的区间存活极限，为下一阶段铺路。

### 1. 区间定义

*   **`First Use Index`**: 在 `sorted_passes` 中该资源第一次被 Read / Write 接触的下标。
*   **`Last Use Index`**: 在 `sorted_passes` 中该资源最后一次参与连线的下标。
*   注意：如果一个资源仅具有 `First = Write` 操作，但压根没有使用（即被上层手动 `force_retain` 保留，比如 Debug View 贴图），它的使用尽头应被拖拽到全图结束 `N` 以确保存活被覆盖。

### 2. 构建任务流：区间收集器实现

*   [ ] 设计 `struct ResourceLifetime { uint32_t first_pass, last_pass; bool is_external; }`。
*   [ ] 用游标遍历平铺的 `sorted_passes` 列表。针对遇到的所有内部引用的 `Resource Edge`：
    *   若是首次遇到当前句柄则设置 `first_pass = current_pass_index`。
    *   一直刷新当前所处下标为 `last_pass = current_pass_index` 至遍历耗尽。
*   [ ] 提供一份带有直观输出的控制台工具/日志导出。将图编译后的 Timeline 和 所有被纳入监控的资源的存活 `[start, end] 线段条` 图形化印出。这是整个基于 Render Graph 构建引擎团队的定海神针与找茬工具。
